# PRD：新闻信号显示优化

## 背景

前端新闻动态页面（News）中，每个币种标签旁显示一个数字，如 `BTC ▲ 960`。用户反馈不清楚该数字的含义，且数值普遍超过 100，与直觉中的"评分/分数"认知不符。

经排查，该数字来自后端接口返回的 `weighted_score` 字段，实际含义是**该币种在指定时间窗口内所有新闻加权分数的累加和**，而非单条新闻的评分。

此外，用户希望将静态数字显示改为**图表化展示**，方便观察信号随时间的变化趋势。

### 当前问题

| 问题 | 说明 |
|------|------|
| 语义不清 | 字段名 `weighted_score` 暗示是"评分"，实际是"累计和" |
| 数值偏大 | 12条新闻 × 每条 ~80分 = 960，用户无法直观理解 |
| 缺少上下文 | 未展示新闻条数，无法判断数字是"强信号"还是"多条叠加" |
| 无归一化 | 不同币种新闻量不同，直接比较累计值不公平 |
| 无法观察趋势 | 静态数字无法展示信号随时间的变化 |

---

## 目标

1. 让用户**一眼看懂**数字代表什么
2. 数值范围回到**直观可比较**的量级（类似 0-100 的评分）
3. 保留**新闻条数**信息，不丢失上下文
4. **图表化展示**信号趋势，方便观察时间维度的变化

---

## 需求

### 功能需求

#### 数字显示优化

| # | 需求 | 说明 |
|---|------|------|
| FR-1 | 显示归一化的平均加权分 | 将 `weighted_score / event_count` 作为主显示值，范围 0-100，可直接比较 |
| FR-2 | 展示新闻条数 | 在 tooltip 或括号内显示 `event_count`，如 `BTC ▲ 80 (12条)` |
| FR-3 | 更新接口返回 | API 新增 `avg_weighted_score` 字段，后端直接计算平均值，避免前端重复计算 |
| FR-4 | 保留原始累计值 | 接口继续返回 `weighted_score`（累计和），供高级分析使用 |

#### 图表可视化

| # | 需求 | 说明 |
|---|------|------|
| FR-5 | 图表位置 | 替换现有 SignalChips，用趋势图完全替代 chip 卡片显示 |
| FR-6 | 时间粒度切换 | 支持小时级（hourly）和天级（daily）视图切换 |
| FR-7 | 自动选择 Top 5 币种 | 默认显示前 5 个最活跃的币种，用户可通过图例切换显示/隐藏 |
| FR-8 | 组合图表显示 | 信号分数用折线图，新闻条数用柱状图，叠加显示 |
| FR-9 | 交互功能 | 支持悬停显示详情、缩放、拖动、图例切换、点击跳转新闻列表 |

### 非功能需求

| # | 需求 | 说明 |
|---|------|------|
| NFR-1 | 向后兼容 | 已有字段不删除，新增字段可选，前端渐进升级 |
| NFR-2 | 中文优先 | UI 文案、注释使用中文 |
| NFR-3 | 国际化 | 新增文案需同步更新 `zh.json` 和 `en.json` |

## 设计方案

### 数字显示方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A：仅显示新闻条数 | 显示 `event_count` 替代加权分 | 最直观，无歧义 | 丢失情绪强度信息 |
| B：显示平均值 | `weighted_score / event_count` | 保留情绪强度，数值回归 0-100 | 需要后端/前端计算 |
| C：平均值 + 条数 | 主显示平均值，tooltip/后缀显示条数 | **信息最完整** | 略复杂 |
| D：保持现状 + 说明 | 加 tooltip 解释"累计加权分" | 改动最小 | 仍未解决数值不可比问题 |

**推荐：方案 C** — 平均值作为主数字，新闻条数作为辅助信息。

### 图表化方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A：单图表 + 时间粒度切换** | 一个主图表，折线（分数）+ 柱状（条数），支持小时/天级切换 | 简洁直观，实现快 | 币种多时可能拥挤 |
| B：双图表 + 详细视图 | 上方折线图（分数），下方柱状图（条数），点击显示新闻列表 | 信息分离清晰，交互直观 | 占用更多空间，实现复杂度中等 |
| C：交互式仪表盘 | 主图表 + 右侧详情面板，拖拽选择时间范围，弹出详细信息 | 功能最强大，用户体验最好 | 实现复杂度最高，可能过度设计 |

**推荐：方案 A** — MVP 优先，快速验证用户需求，后续根据反馈迭代。

**后续迭代路径**：
- V1：方案 A（快速上线）
- V2：根据用户反馈，决定是否升级到方案 B 或 C
